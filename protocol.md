# WinkYou 自研协议设计方案

> 本文档探讨：如果要从协议层获得极致性能，需要动什么、怎么动、有什么代价。
> 
> 前提：团队有密码学背景，能理解协议安全性分析，愿意承担自研协议的风险。
>
> 创建日期: 2026-04-02

## 一、为什么按WireGuard协议实现打不过wireguard-go

先把这个问题彻底说清楚。

**协议固定开销 — 谁都跑不掉：**

```
每个数据包的固定计算量:

ChaCha20加密:  ~4 cycles/byte (软件实现)
Poly1305认证:  ~1.5 cycles/byte
包头处理:       固定开销

1400字节的包:
  加密: 1400 × 4 = 5600 cycles
  认证: 1400 × 1.5 = 2100 cycles
  总计: ~7700 cycles + 包头处理

10Gbps线速 = 每秒约90万个1400字节包
每包7700 cycles → 需要约7GHz的单核 (不现实)
所以高吞吐必须:
  - 多核并行
  - 批量处理
  - 硬件加速
```

**wireguard-go已经做的优化：**
- 使用汇编优化的ChaCha20-Poly1305（x/crypto里的）
- 批量处理（尽可能一次系统调用收发多个包）
- 连接池、缓冲池

**你按同样协议实现能额外做的：**
- 垂直整合省掉内存拷贝（10-20%）
- 去掉UAPI等不需要的功能（减小代码路径）
- 更激进的批量策略

**上限：比wireguard-go快10-30%，不可能翻倍。**

## 二、协议层可以动的地方

要数量级的提升，必须改协议。下面是每个可以动的点，以及代价。

### 2.1 密码套件选择

**现状**: WireGuard固定使用ChaCha20-Poly1305

**问题**: 
- 在有AES-NI的x86 CPU上，AES-GCM比ChaCha20快2-3倍
- 服务器、PC大多有AES-NI
- ARM手机上ChaCha20更快

**优化方案**:

```
协商密码套件:
  - 握手时交换支持的密码套件列表
  - 选择双方都支持的最快的那个
  - 可选: ChaCha20-Poly1305, AES-128-GCM, AES-256-GCM

性能对比 (现代CPU实测, ashvardanian.com 2025):
  ChaCha20-Poly1305:  467-1580 MB/s
  AES-256-GCM:        862-3731 MB/s (AES-NI)
  提升:               ~2-3倍

实现复杂度: 低
  - 握手消息加一个字段
  - 运行时根据套件调用不同的加密函数
  
安全代价: 几乎无
  - AES-GCM和ChaCha20-Poly1305安全性相当
  - 都是AEAD，都经过广泛分析
```

**这是最值得做的优化——低风险高回报。**

### 2.2 握手简化

**现状**: WireGuard的Noise IK握手
- 4次DH操作
- 2条消息
- 前向保密 + 身份隐藏

**可以砍的**:

```
场景1: 预共享密钥模式 (PSK-only)

如果双方事先共享了密钥，可以跳过整个握手:
  - 直接用PSK派生传输密钥
  - 0条握手消息，直接发数据

适用: 数据中心内部、边缘节点、物联网设备
代价: 
  - 失去前向保密（PSK泄露则历史流量可解密）
  - 失去身份动态认证

实现复杂度: 低
  KDF(PSK, nonce) → (send_key, recv_key)
  每个包带一个递增nonce，足矣


场景2: 简化Noise握手

IK模式: 4次DH
  DH(i_e, r_s), DH(i_s, r_s), DH(i_e, r_e), DH(i_s, r_e)

如果不需要身份隐藏，可以用Noise XX:
  3条消息，但每条消息更小

或者自定义握手:
  只做 DH(i_e, r_e) + DH(i_s, r_s)
  省掉2次DH，约快一倍

代价: 
  - 可能降低安全性
  - 需要密码学分析确认

实现复杂度: 中
```

### 2.3 去掉每包重放保护

**现状**: WireGuard每个包检查重放窗口（滑动位图）

**成本**: 每包一次窗口查询和位操作

**可以砍掉的场景**:

```
可信网络内部:
  - 数据中心内网（没有中间人）
  - 点对点专线

不可砍的场景:
  - 公网传输
  - 多跳路由

实现:
  配置项: replay_protection = true/false
  false时跳过窗口检查

代价:
  - 重放攻击可行（但AEAD保证不能篡改）

性能提升: 很小 (~1-2%)
  - 窗口检查本身很快
  - 不是瓶颈
```

这个优化收益太小，不值得。

### 2.4 批量加密

**现状**: 每个IP包单独加密

**优化思路**:

```
方案A: GSO式批量

接收端:
  攒多个小包 → 拼成一个大buffer → 一次加密调用

发送端:
  一个大密文 → 拆成多个UDP包发送

问题:
  - 需要接收端等待（增加延迟）
  - 拆包重组复杂
  - 丢包时整批重传


方案B: 并行加密

多个包并行加密（利用多核）:
  包1 → goroutine 1 → 加密
  包2 → goroutine 2 → 加密
  包3 → goroutine 3 → 加密

问题:
  - 需要维护包序
  - goroutine调度开销
  - 对小包效果差


方案C: SIMD并行 (最有价值)

用AVX-512一次处理多个ChaCha20块:
  - 一次指令加密8个64字节块
  - 吞吐提升约4倍（vs AVX2）

问题:
  - 需要汇编
  - Go目前对AVX-512支持不好
  - 或用cgo调用C实现
```

**方案C是真正能带来数量级提升的，但需要深入底层。**

### 2.5 零拷贝DMA (终极优化)

**现状**: 
```
网卡 → 内核buffer → 用户态buffer → 解密 → TUN buffer → 内核 → 应用
    拷贝         拷贝              拷贝          拷贝
```

**终极形态**:
```
网卡 → 用户态buffer (DMA) → 解密(原地) → TUN (sendfile) → 应用
    零拷贝               无拷贝        零拷贝
```

**需要的技术**:
- AF_XDP (Linux 4.18+) — 网卡直接DMA到用户态
- io_uring — 异步系统调用
- sendfile/splice — 零拷贝到TUN

**复杂度**: 极高
**平台限制**: 仅Linux
**性能提升**: 可能2-5倍（系统调用和拷贝是大头）

## 三、自研协议的安全性如何保证

改协议最大的风险是引入安全漏洞。必须有系统方法验证安全性。

### 3.1 方法论

```
层次1: 密码学原语安全

只使用经过验证的原语:
  - AES-GCM, ChaCha20-Poly1305 — 标准AEAD
  - Curve25519, X25519 — 标准ECDH
  - BLAKE2, SHA-256 — 标准哈希
  - HKDF — 标准KDF

不要:
  - 自己设计密码算法
  - 魔改标准算法的参数
  - 使用非标准的组合方式


层次2: 协议安全

需要分析的属性:
  - 前向保密 (Forward Secrecy)
  - 密钥确认 (Key Confirmation)
  - 身份隐藏 (Identity Hiding)
  - 抗重放 (Replay Protection)
  - 抗中间人 (MITM Protection)

分析方法:
  - 手工推导（写出协议流程，分析每种攻击）
  - 符号化验证（ProVerif, Tamarin）
  - 请外部专家审计


层次3: 实现安全

  - 时间恒定比较（防止timing attack）
  - 内存安全（Go比C好，但仍需注意cgo部分）
  - 随机数质量（crypto/rand）
  - 密钥清理（用完置零）
```

### 3.2 哪些简化是"安全"的

```
安全的简化:
✅ 添加AES-GCM作为可选密码套件（和ChaCha20同等安全）
✅ 减少握手消息大小（只要该有的字段都有）
✅ 批量加密多个包（只是性能优化，不影响安全性）

需要仔细分析的简化:
⚠️ 减少DH次数（可能降低身份隐藏或前向保密）
⚠️ 延长密钥生命周期（增加密钥泄露的窗口）
⚠️ 去掉timestamp（可能允许重放旧握手）

危险的简化:
❌ 去掉AEAD的认证标签（完全失去完整性保护）
❌ 使用ECB模式或其他非认证加密
❌ 复用nonce（灾难性）
```

### 3.3 协议演进策略

```
阶段1: WireGuard兼容层

保持协议完全兼容WireGuard，只做实现优化:
  - 可以和原版WireGuard互通
  - 零协议风险

阶段2: 协商扩展

在握手中加入扩展字段:
  - 协商密码套件（AES-GCM等）
  - 协商特性开关
  - 对不支持扩展的peer回退到标准模式

阶段3: 新协议模式

引入新的协议模式（和WireGuard不兼容）:
  - PSK直连模式
  - 低延迟模式
  - 通过配置选择使用哪种模式

这样做的好处:
  - 渐进式风险
  - 保持和生态的兼容性
  - 新模式可以先在内部测试
```

## 四、具体协议设计方案

下面是一个具体的自研协议设计，代号 **Wink Protocol v1**。

### 4.1 设计目标

```
必须保证:
  - 端到端加密
  - 前向保密（至少对数据，握手可选）
  - 抗中间人
  - 身份认证

可以放松:
  - 身份隐藏（不是所有场景都需要）
  - 抗重放（可信网络可选关闭）

性能目标:
  - 比WireGuard快2倍（有AES-NI时）
  - 握手延迟减半
```

### 4.2 密码套件

```
定义两个套件:

WINK_CHACHA20_POLY1305:
  - 握手: X25519 + BLAKE2s
  - 数据: ChaCha20-Poly1305
  - 适用: ARM、无AES-NI的设备

WINK_AES128_GCM:
  - 握手: X25519 + SHA-256 (AES-NI也加速SHA)
  - 数据: AES-128-GCM
  - 适用: x86服务器、PC

协商:
  - Initiator在握手消息1中列出支持的套件（按优先级）
  - Responder选择自己也支持的最高优先级套件
  - 在消息2中确认选择
```

### 4.3 握手协议

```
基于Noise KK模式（双方公钥预知，最终设计见 wink-protocol-v1.md）:

> 注意: 本节早期探索使用了NK模式，经深入分析后最终选择了KK模式。
> KK模式下双方静态公钥都预知，无需加密传输静态公钥，
> 消息1从148字节减至96字节。完整字节级格式见 wink-protocol-v1.md。

消息1: Initiator → Responder
┌─────────────────────────────────────────────────────────────┐
│ type (1B) │ suites (1B) │ sender_idx (2B) │ ephemeral (32B) │
│ encrypted_timestamp (28B) │ mac1 (16B) │ mac2 (16B)         │
└─────────────────────────────────────────────────────────────┘

  1. Initiator生成临时密钥 (e_i)
  2. DH(e_i, responder_static) → 混入链式密钥 (es)
  3. DH(initiator_static, responder_static) → 混入链式密钥 (ss, 预计算)
  4. 加密payload: timestamp (KK模式无需加密静态公钥)
  5. 发送


消息2: Responder → Initiator
┌─────────────────────────────────────────────────────────────┐
│ version (1B) │ suite (1B) │ ephemeral_pub (32B) │           │
│ encrypted_payload (16B)                                     │
└─────────────────────────────────────────────────────────────┘

  1. Responder生成临时密钥 (e_r)
  2. DH(e_r, initiator_ephemeral) → shared2
  3. DH(responder_static, initiator_static) → shared3
  4. 传输密钥: (send, recv) = HKDF(shared1 || shared2 || shared3, "wink-data")
  5. 加密空payload作为确认
  6. 发送

完成: 双方得到 (send_key, recv_key)

DH次数: 3次 (vs WireGuard的4次)
消息: 2条 (和WireGuard相同)
前向保密: 有（ephemeral密钥提供）
```

### 4.4 数据传输

```
数据包格式:
┌─────────────────────────────────────────────────────────────┐
│ receiver (4B) │ counter (8B) │ encrypted_payload (N+16B)   │
└─────────────────────────────────────────────────────────────┘

和WireGuard几乎相同，但:
  - 去掉1字节type字段（数据包固定类型）
  - 节省每包1字节

加密:
  AEAD(key, nonce=counter, plaintext=IP包, ad=receiver||counter)

为什么ad包含receiver和counter:
  - 防止跨连接重放
  - 防止counter篡改
```

### 4.5 PSK直连模式

```
适用场景:
  - 数据中心内部
  - 配置好的设备对
  - 延迟敏感场景

流程:
  1. 双方预配置共享的PSK
  2. 连接时:
     send_key = HKDF(PSK, local_id || remote_id || "send")
     recv_key = HKDF(PSK, local_id || remote_id || "recv")
  3. 直接发数据，无握手

安全性:
  - 无前向保密（PSK泄露则所有流量可解密）
  - 仍有认证（不知道PSK无法伪造）
  - 仍有加密（流量保密）

性能:
  - 握手延迟: 0
  - 首包延迟: 1 RTT（发出去就是数据）
```

### 4.6 特性对比

| 特性 | WireGuard | Wink v1 (Full) | Wink v1 (PSK) |
|------|-----------|----------------|---------------|
| 握手消息数 | 2 | 2 | 0 |
| 握手DH次数 | 4 | 3 | 0 |
| 前向保密 | ✓ | ✓ | ✗ |
| 身份隐藏 | ✓ | ✗ | ✗ |
| 密码套件 | 固定 | 可协商 | 可协商 |
| 首包延迟 | 1 RTT | 1 RTT | 0 RTT |
| AES-NI加速 | ✗ | ✓ | ✓ |

## 五、实现路线

```
第一步: 在现有框架上加密码套件协商
═══════════════════════════════════════════════
  - 握手消息加suites字段
  - 实现AES-GCM数据加密
  - 和自己的旧版本互通测试
  
  工作量: ~500行
  风险: 低
  收益: 有AES-NI时数据加密快2-3倍

第二步: 简化握手
═══════════════════════════════════════════════
  - 实现3次运行时DH的Noise KK模式（最终设计见 wink-protocol-v1.md）
  - 去掉身份隐藏换取更快握手
  - 安全性分析

  工作量: ~400行
  风险: 中（需要验证协议安全性）
  收益: 握手快25%

第三步: PSK直连模式
═══════════════════════════════════════════════
  - 实现0-RTT PSK模式
  - 配置文件支持PSK
  - 文档说明安全取舍

  工作量: ~300行
  风险: 低（PSK模式的安全属性很清晰）
  收益: 延迟敏感场景首包无等待

第四步: SIMD加密优化
═══════════════════════════════════════════════
  - 引入汇编/cgo的AES-GCM实现
  - 或使用 Intel的 isa-l_crypto
  - 批量加密接口

  工作量: ~800行 或 集成现有库
  风险: 中（汇编代码维护难）
  收益: 数据吞吐再提升2-4倍

第五步: 零拷贝IO (Linux专项)
═══════════════════════════════════════════════
  - AF_XDP集成
  - io_uring集成
  - 需要Linux 5.x+

  工作量: ~1500行
  风险: 高（和内核版本绑定）
  收益: 系统调用开销趋近于零
```

## 六、风险与缓解

### 协议安全风险

```
缓解措施:
1. 分阶段引入修改，每阶段做安全审计
2. 新协议模式在内部长时间测试后再开放
3. 提供配置选项回退到标准WireGuard协议
4. 邀请外部安全研究者审计
5. 运行fuzzer长期测试
```

### 性能提升不及预期

```
缓解措施:
1. 每一步优化都单独benchmark
2. 如果某一步提升不明显，果断放弃
3. 80%的收益来自20%的改动（密码套件选择是大头）
```

### 维护复杂度增加

```
缓解措施:
1. 明确的协议版本号和特性标志
2. 每种模式独立的代码路径
3. 完善的测试矩阵（所有模式组合）
```
