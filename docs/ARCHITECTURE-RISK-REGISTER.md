# WinkYou 架构改造风险清单

> **目的**: 系统性识别架构改造中的风险，制定缓解措施和回滚方案
> 
> **更新频率**: 每周一次

---

## 风险评级体系

### 风险等级

| 等级 | 描述 | 行动 |
|------|------|------|
| 🔴 严重 | 可能导致系统不可用，数据丢失 | 必须缓解，每日跟踪 |
| 🟠 高 | 影响核心功能或性能 | 必须缓解，每周跟踪 |
| 🟡 中 | 影响部分功能或可维护性 | 建议缓解，监控 |
| 🟢 低 | 影响范围有限 | 可接受，记录 |

### 概率评级

- **极高 (>70%)**: 几乎肯定发生
- **高 (40-70%)**: 很可能发生
- **中 (20-40%)**: 可能发生
- **低 (<20%)**: 不太可能发生

## 风险清单

### R001: 细粒度锁引入并发 Bug 🔴

**描述**: 将 engine 的大锁拆分为细粒度锁，可能引入死锁、竞态条件

**关联任务**: 改进方案 01

**概率**: 中 (30%)  
**影响**: 严重 - 可能导致生产环境崩溃

**触发条件**:
- 多个细粒度锁的锁顺序不一致
- 读写锁使用不当
- atomic 操作和锁混用

**缓解措施**:
1. ✅ 严格的 Code Review，至少 2 人审核
2. ✅ 每个 PR 必须通过 `go test -race`
3. ✅ 编写并发测试用例（至少 10 个）
4. ✅ 使用 deadlock detector
5. ✅ 在测试环境运行 7 天压力测试

**回滚方案**:
- 保留旧代码作为 backup branch
- Feature flag 控制新旧代码切换
- 监控异常时 5 分钟内可切回

**责任人**: 后端核心-1  
**状态**: 监控中

---

### R002: 状态机重构破坏现有流程 🟠

**描述**: 引入严格的状态转换验证，可能拒绝某些合法但未识别的转换

**关联任务**: 改进方案 02

**概率**: 高 (50%)  
**影响**: 高 - 部分场景下连接失败

**触发条件**:
- 边缘案例（重连、降级）的状态转换未在白名单
- 时序问题导致状态转换被拒绝

**缓解措施**:
1. ✅ 详细分析现有状态流，列出所有合法转换
2. ✅ 收集 1 周的生产环境状态转换数据
3. ✅ 初期使用警告模式（记录非法转换但不拒绝）
4. ✅ 灰度发布，10% → 50% → 100%

**回滚方案**:
- 保留旧状态机实现
- 通过配置切换新旧状态机
- 异常率超过 1% 自动回滚

**责任人**: 后端核心-1  
**状态**: 待启动

---

### R003: 错误处理大规模迁移引入回归 🟠

**描述**: 改造涉及全代码库，可能引入隐藏的逻辑错误

**关联任务**: 改进方案 04

**概率**: 高 (60%)  
**影响**: 高 - 错误处理逻辑变化

**触发条件**:
- 错误包装改变了 errors.Is/As 行为
- 上下文信息丢失
- 重试逻辑判断错误

**缓解措施**:
1. ✅ 分模块迁移（先 session，再 solver，最后 client）
2. ✅ 保留原始错误链（不破坏 errors.Is）
3. ✅ 全面的错误处理测试用例
4. ✅ 灰度发布

**回滚方案**:
- 单模块出问题时只回滚该模块
- 保留 v1 错误处理函数

**责任人**: 后端工具  
**状态**: 待启动

---

### R004: 性能优化导致功能回归 🟠

**描述**: 内存池、零拷贝等优化可能引入隐蔽的内存安全问题

**关联任务**: 改进方案 06、09

**概率**: 中 (35%)  
**影响**: 高 - 数据损坏或崩溃

**触发条件**:
- 对象池中对象未正确重置
- 零拷贝缓冲区被并发修改
- 引用计数错误

**缓解措施**:
1. ✅ 使用 race detector 全程开启
2. ✅ 每个池化对象增加 `validate()` 方法
3. ✅ Debug 模式下启用对象生命周期追踪
4. ✅ 模糊测试（fuzzing）

**回滚方案**:
- 池化对象通过 feature flag 切换
- 异常时降级为直接分配

**责任人**: 后端工具、后端核心-2  
**状态**: 待启动

---

### R005: 分布式追踪性能开销超预期 🟡

**描述**: OpenTelemetry 集成可能带来明显性能开销

**关联任务**: 改进方案 05

**概率**: 中 (40%)  
**影响**: 中 - 性能下降 5-10%

**触发条件**:
- 采样率配置过高
- Span 数量爆炸
- Exporter 阻塞主流程

**缓解措施**:
1. ✅ 默认采样率 10%（生产环境）
2. ✅ Async exporter，不阻塞业务
3. ✅ 关键路径基准测试
4. ✅ 可通过环境变量动态调整

**回滚方案**:
- 通过环境变量关闭追踪
- 不影响业务代码

**责任人**: DevOps  
**状态**: 待启动

---

### R006: 协议版本化破坏向后兼容 🔴

**描述**: V2 协议可能与 V1 不兼容，影响存量用户

**关联任务**: 改进方案 08

**概率**: 中 (30%)  
**影响**: 严重 - 老用户无法使用

**触发条件**:
- V2 字段名/类型与 V1 冲突
- 协商逻辑错误
- 降级路径有 bug

**缓解措施**:
1. ✅ V1 客户端能成功连接 V2 服务端
2. ✅ V2 客户端能成功连接 V1 服务端
3. ✅ 兼容性矩阵测试（V1/V2 × V1/V2）
4. ✅ 老版本至少保留 6 个月

**回滚方案**:
- V2 协议默认关闭
- 配置开关启用 V2

**责任人**: 架构师  
**状态**: 待启动

### R007: Goroutine 池限流影响业务 🟡

**描述**: 限流配置过严可能导致正常请求被拒绝

**关联任务**: 改进方案 07

**概率**: 中 (40%)  
**影响**: 中 - 部分请求失败

**触发条件**:
- MaxConcurrent 设置过低
- 突发流量超过容量

**缓解措施**:
1. ✅ 初期设置较高的 MaxConcurrent
2. ✅ 监控被拒绝的请求数
3. ✅ 动态调整能力
4. ✅ 优雅降级机制

**回滚方案**:
- 设置 MaxConcurrent = MaxInt（无限制）
- 不影响功能

**责任人**: 后端工具  
**状态**: 待启动

### R008: 混沌测试基础设施延迟 🟡

**描述**: 混沌测试环境搭建复杂，可能影响 Phase 4 进度

**关联任务**: 改进方案 10

**概率**: 高 (50%)  
**影响**: 中 - 延期 1-2 周

**触发条件**:
- 网络故障注入需要 root 权限
- CI 环境隔离要求高
- 测试场景设计复杂

**缓解措施**:
1. ✅ Phase 1 就开始准备测试环境
2. ✅ 简单场景先做（用 Go 代码模拟）
3. ✅ 复杂场景（kernel-level）后做
4. ✅ 找有经验的工程师参与

**回滚方案**:
- 推迟到 Phase 4 之后单独立项
- 不影响其他改造

**责任人**: 测试工程师  
**状态**: 待启动

## 更新机制

### 周更新

每周一站会同步：
- 本周新识别的风险
- 已缓解的风险
- 触发的风险及处理情况

### 月度评审

每月底全量评审：
- 风险清单是否完整
- 缓解措施是否有效
- 是否需要调整等级

### 触发更新

以下情况立即更新清单：
- 新风险被识别
- 风险等级变化
- 缓解措施失效
- 回滚被触发
