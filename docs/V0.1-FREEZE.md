# v0.1 Freeze Gate

This document defines what must be true before cutting the first v0.1 Alpha release.

## Status

v0.1 is the self-hostable alpha milestone. It is not a feature-complete consumer VPN product.

Active architecture remains:

```text
WinkYou = connectivity solver + secure WireGuard data plane
```

The authority for connectivity/session/strategy/transport boundaries remains [`CONNECTIVITY-SOLVER-BASELINE.md`](./CONNECTIVITY-SOLVER-BASELINE.md).

## Current Architecture

- Rendezvous and coordinator handle node registration, peer discovery, and session message exchange.
- `pkg/session` owns session lifecycle, state transitions, capability exchange, strategy invocation, observation/probe/path commit envelopes, and binder coordination.
- `pkg/solver` stays strategy-agnostic.
- Strategy-specific behavior lives under `pkg/solver/strategy/*` or the client assembly edge.
- `transport.PacketTransport` is the stable packet boundary.
- `pkg/tunnel` consumes packet transports through userspace WireGuard and does not choose paths.
- TURN relay is a valid solved path, not a failure state.

## Completed Capabilities

- Linux coordinator can be built and run.
- coturn quickstart materials are available for TURN relay.
- Two clients can join one virtual network through the documented self-host flow.
- `legacy_ice_udp` remains the compatibility strategy for direct-prefer ICE/UDP.
- `relay_only` can be selected explicitly with `connectivity.mode: relay_only`.
- Empty remote capability from old peers still falls back to `legacy_ice_udp`.
- Ordered strategy fallback exists.
- Scoped observations can influence strategy ordering in `auto` mode.
- `tcp_framed` exists as an alpha non-UDP PacketTransport proof using explicit reachable TCP addresses.
- `wink doctor` provides layered diagnostics.
- `wink logs` can read configured file logs.
- Long-running client workflow is documented for Linux systemd and Windows Task Scheduler/NSSM.
- Release workflow builds client/coordinator/relay artifacts and `SHA256SUMS`.

## Supported Platforms

Server-side:

- Linux coordinator
- Linux coturn deployment
- Linux relay helper binary

Client-side:

- Windows amd64 client with administrator privileges and Wintun/TUN support
- Linux amd64 client with `/dev/net/tun` and root or equivalent network capabilities

CI and tests also use memory backends where privileged TUN is not available.

## Supported Strategies

Stable or compatibility strategies:

- `legacy_ice_udp`
- `relay_only`

Alpha strategy:

- `tcp_framed`

`tcp_framed` is intentionally alpha. It requires an explicitly reachable TCP address and does not claim NAT TCP hole punching.

## Deployment Path

Primary self-host path:

1. Start coordinator and coturn from [`deploy/quickstart`](../deploy/quickstart).
2. Generate two client configs from quickstart templates.
3. Start both clients with `wink up`.
4. Check `wink status` and `wink peers`.
5. Verify WireGuard handshake and ping over virtual IP.
6. Switch both nodes to `connectivity.mode: relay_only`.
7. Verify relay path with `wink peers` and `wink doctor --relay`.

## Regression Gate

Before cutting v0.1, run:

```bash
go test ./... -count=1
go test -race ./pkg/session ./pkg/client ./pkg/solver/... -count=1
make test-phase2d
make test-phase3a
make test-phase4a
make build-all
```

If the local Windows shell does not have `make`, use the w64devkit/MSYS2 shell or add its `bin` directory to `PATH` for that command. Race tests also require cgo/GCC.

## Release Gate

The v0.1 release workflow must produce:

- `wink-windows-amd64.exe`
- `wink-linux-amd64`
- `wink-coordinator-linux-amd64`
- `wink-relay-linux-amd64`
- `SHA256SUMS`

Each binary should report the expected version through:

```bash
wink-linux-amd64 version
wink-coordinator-linux-amd64 --version
wink-relay-linux-amd64 --version
```

The Windows client should report version through:

```powershell
wink-windows-amd64.exe version
```

## Known Limits

- No GUI.
- No mobile client.
- No no-admin mode.
- No native Windows service binary.
- No QUIC datagram strategy.
- No HTTP CONNECT or WebSocket fallback strategy.
- No self-developed Wink Protocol data plane.
- `tcp_framed` remains alpha.
- Advanced learning/scoring is not a v0.1 requirement; current scoring only adjusts ordering from scoped recent observations.
- The self-host quickstart must still be validated in a real public-server environment before tagging.
- Established direct paths can still be disrupted by coordinator/control-plane loss in current client behavior; see [`CONTROL-PLANE-RESILIENCE.md`](./CONTROL-PLANE-RESILIENCE.md).
- Pure NAT-piercing validation still needs ICE candidate interface filtering to exclude existing overlays such as Tailscale, Docker bridge, or other VPN/TAP interfaces.

## v0.2 Roadmap

v0.2 should focus on hardening rather than expanding the architecture:

- Preserve already-bound data paths during temporary coordinator/control-plane loss.
- Add status fields and tests that distinguish control-plane health from data-plane health.
- Add ICE candidate interface filtering so pure NAT-piercing validation can exclude Tailscale, Docker bridge, and other VPN/TAP interfaces.
- Promote or replace `tcp_framed` based on real LAN/explicit-TCP testing.
- Add a proxy-friendly transport proposal or implementation after v0.1.
- Improve strategy-order observability so `wink doctor` can explain why an order changed.
- Broaden regression coverage across `legacy_ice_udp`, `relay_only`, and `tcp_framed`.
- Decide whether daemon/service support needs native Windows service integration.
- Keep QUIC datagram and self-developed data-plane work in proposal scope until there is a measured need.
